# Razad

> *Your server, guided.* — Server kamu, jalur yang benar.

**Razad** is an open-source server management and deployment platform built for solo developers and small teams that deploy applications on Linux VPS or dedicated servers. It combines lightweight native process management, automated app deployment, database provisioning, domain and SSL management, real-time logs, and a proactive AI layer that can observe server state and safely execute approved non-destructive actions.

No Docker required. No Kubernetes. Just a single Go binary, systemd, and Nginx.

---

## Features

- **App Management** — Deploy apps from Git repos or uploads. Auto-detect runtimes (Node.js, Go, Python, PHP, Ruby, Bun). Start, stop, restart, and manage environment variables (encrypted at rest).
- **Process Management** — Native systemd integration. Generate service units automatically. Auto-restart on crash. Survives server reboots.
- **Database Provisioning** — Provision PostgreSQL, MySQL, and Redis instances in one step. Handle credentials securely.
- **Domain & SSL** — Bind custom domains to apps. Auto-generate Nginx config. Issue SSL certificates via Let's Encrypt/Certbot.
- **Log Streaming** — Real-time logs via WebSocket. Filter by app, severity, and time range.
- **AI Assistant** — Proactive server monitoring with natural-language interaction. BYOK support (OpenAI, Anthropic, Gemini, Ollama). Strictly whitelisted, non-destructive actions only.
- **Audit Trail** — Immutable audit logging for all user and AI actions.
- **Self-Healing** — AI can detect crashed apps and trigger approved recovery flows.

## Architecture

### Two-Plane Design

Razad separates into two planes — this is the primary scalability mechanism:

```
┌──────────────────────────────────────────────────┐
│              Control Plane (Cloud)               │
│  Identity · Dashboard · AI · Billing · Fleet     │
└──────────────────────┬───────────────────────────┘
                       │ Agent Protocol (secure)
┌──────────────────────▼───────────────────────────┐
│              Execution Plane (Node)               │
│  systemd · Nginx · Apps · DBs · Logs · Configs   │
└──────────────────────────────────────────────────┘
```

- **Execution Plane** — runs on the customer server. Manages processes, configs, networking, and logging. The node is the authority for all local system changes.
- **Control Plane** — runs in Razad Cloud (optional). Provides the dashboard, AI, multi-tenant auth, billing, and fleet visibility. Cloud services request actions; the node enforces them.

### Three Operating Modes

1. **Self-Hosted** — single binary, no cloud dependency. Everything runs on your server.
2. **BYO VPS Cloud** — you own the VPS, Razad Cloud manages it remotely.
3. **Managed Infrastructure** — Razad provisions and operates infrastructure for you.

### Tech Stack

| Layer | Technology |
|---|---|
| Backend | Go (single binary, statically compiled) |
| Frontend | SvelteKit + TypeScript (embedded in Go binary) |
| Process Supervisor | systemd (native Linux) |
| Reverse Proxy | Nginx (config generated by Razad) |
| Databases | PostgreSQL (primary), SQLite (optional self-hosted) |
| Messaging | Redis, NATS |
| AI Providers | OpenAI, Anthropic, Google Gemini, Ollama (BYOK) |
| SSL | Certbot / Let's Encrypt |
| License | AGPLv3 |

## Quick Start (Development)

### Prerequisites

- Go 1.22+
- Node.js 20+
- CGO_ENABLED=1 (required for SQLite support)

### Setup

```bash
# Clone the repository
git clone https://github.com/razad/razad.git
cd razad

# Install Go dependencies
go mod tidy

# Install frontend dependencies
make web-setup

# Build frontend assets
make web-build

# Build the daemon binary
make build

# Run tests
make test
```

### Development Mode

```bash
# Terminal 1: Start the backend
make dev

# Terminal 2: Start the frontend with hot-reload
make web-dev
```

The Vite dev server proxies `/api` requests to the Go backend on `:8080`.

### Full Release Build

```bash
make release
```

This builds the SvelteKit frontend, embeds it into the Go binary, and outputs a single static binary to `build/razad-daemon`.

### Makefile Targets

| Target | Description |
|---|---|
| `build` | Build the daemon binary |
| `run` | Build and run locally |
| `dev` | Run in development mode |
| `test` | Run all unit tests |
| `lint` | Run golangci-lint |
| `fmt` | Format and vet Go code |
| `web-setup` | Install frontend dependencies |
| `web-build` | Build frontend static assets |
| `web-dev` | Start Vite dev server |
| `release` | Full release build (frontend + daemon) |

## Architecture Principles

| Principle | Meaning |
|---|---|
| **Native-first** | systemd, Nginx, and local services are first-class citizens |
| **Lightweight** | Single binary, no Docker, minimum runtime overhead |
| **Safe automation** | AI is restricted to an explicit whitelist of non-destructive actions |
| **Audit everything** | Every privileged action, especially AI-driven, is immutably logged |
| **Fail closed** | When validation or policy evaluation fails, the request is denied |
| **Boundary-driven** | Execution plane and control plane have clear, enforced boundaries |
| **Local authority** | The node is the authority for local changes — cloud requests, node enforces |

---

## AI Safety

Razad's AI assistant is designed with safety as a core architectural constraint:

- **Whitelist-only execution** — AI may only perform actions in the explicit registry
- **Blocked actions** — delete app, drop database, modify firewall, execute arbitrary shell, modify production env vars
- **Audit trail** — every AI proposal, policy decision, and execution result is recorded
- **BYOK** — bring your own API key (OpenAI, Anthropic, Gemini, Ollama)

## Contributing

### Branch Strategy

- **Trunk-based development** with short-lived feature branches off `main`
- Branch naming: `feature/<name>`, `fix/<name>`, `chore/<name>`, `hotfix/<name>`, `refactor/<name>`
- No direct commits to `main` — everything goes through review

### Commit Convention

```
type(scope): description
```

Types: `feat`, `fix`, `refactor`, `chore`, `docs`, `test`, `ci`, `perf`, `sec`

Examples:
- `feat(app): add deployment start flow`
- `fix(proxy): validate nginx config before reload`
- `docs(readme): update development setup`

### Pull Request Checklist

- [ ] Code builds (`make build`)
- [ ] Tests pass (`make test`)
- [ ] Linter passes (`make lint`)
- [ ] Behavior is documented
- [ ] Security-sensitive changes are reviewed
- [ ] Migration changes are safe (if applicable)

### Testing Requirements

- Every module must have basic unit coverage
- Critical flows require integration tests
- Security-sensitive logic requires negative tests
- Bug fixes must include regression tests
